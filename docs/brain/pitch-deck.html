I · The Premise
Signals → Ships
The pipeline, declared.

Everything we build is a projection of what we've heard.

01 signals raw evidence · unbounded append-only
02 project memory inspectable canonical truth · compiled pages + sources
03 opportunity clusters the long backlog · 50–200 themed bets
04 ranked roadmap focus-weighted slice · 10–25 reviewable
05 PRDs / build jobs execution shortlist · 1–5 active

Two new layers do the work that roadmap_items and prd_content were quietly overloaded with: memory that lives between signals and artifacts, and a resolver that decides where anything goes next.

signals → project memory → opportunity clusters → ranked roadmap → PRDs / build jobs

II · Post-Mortem
Three failures we're retiring
What V1 got wrong.

The backlog became unusable because three roles collapsed into one row.

Failure 01

Roadmap rows were doing memory, output, and queue.

roadmap_items and prd_content were simultaneously the durable record of what we believe, the generated artifact, and the action queue. No layer could be fixed without touching the others.

Failure 02

Every signal batch minted a new brief.

A capped ranked roadmap coexisted with an ever-growing briefs layer. Repeated evidence created duplicate opportunities instead of strengthening existing ones. The backlog degraded into a junk drawer.

Failure 03

Skills invented their own filing logic.

No one knew where evidence was supposed to land. Capabilities went dark — the skill existed, but nothing could reach it. Routing drifted silently.

v1.1 separates memory, backlog, and execution — with governance between them

IV · Principles
Five clauses · the constitution
Principles.

The constitution each skill is expected to obey.

§ 01 Thin harness. Fat skills.

The app resolves context, applies guardrails, records runs, and writes structured outputs. Judgment lives in reusable markdown procedures.

§ 02 Deterministic stays deterministic.

Ranking math, focus weighting, dedup, resolver selection, and run logging run in code and SQL. The model never does arithmetic.

§ 03 Memory is inspectable.

Canonical truths live in pages and page versions with citations back to the source evidence. History is append-only.

§ 04 Routing is explicit.

No skill invents its own invocation path. Triggers, priorities, and fallbacks live in tables the audit can read.

§ 05 No hidden filing.

Before creating a new page or cluster, every memory-writing skill reads RESOLVER.md and the filing rules.

If the same judgment is needed twice, it becomes a rule.

V · Resolvers compose
Four layers, one discipline
The resolver is not one thing.

Routing is a system — four composable layers, each with its own failure mode.

I

Skill Resolver

agent config + resolver_triggers

Routes user requests, cron events, and webhooks to the right skill. Keeps capabilities reachable and prevents overlap between similar skills.

the skill exists, but nobody can invoke it.

II

Filing Resolver

docs/brain/RESOLVER.md + _filing-rules.md

Decides whether new evidence updates a page, strengthens a cluster, creates a new cluster, or queues a review. Keeps memory MECE enough to stay usable.

every skill invents its own filing logic. memory becomes a junk drawer.

III

Context Resolver

brain_resolver_rules

Loads the smallest correct context set for a task. Puts current_focus first for roadmap work. Enforces required vs. optional pages by priority.

giant prompts, stale assumptions, noisy context.

IV

Action Resolver

runtime policy + approval rules

Turns a cluster change into the right next action: rerank, refresh a brief, draft a PRD, queue implementation, or ask for approval.

evidence leaks straight to execution with no gate.

Every new skill must declare its triggers. Every memory-writing skill must read the resolver first.

VI · The Backlog
A four-tier funnel with clear update policy
The backlog model.

A long backlog is fine — if each tier has a job.

01 Signals Raw evidence from feedback, analytics, support, scans, and shipped outcomes. ∞unbounded never rank directly · route through filing
02 Opportunity clusters The canonical long backlog. Grouped by theme and dominant need. 50 — 200active default action is attach · minting is rare
03 Ranked roadmap A focus-weighted slice of the long backlog that a team can actually review. 10 — 25live recomputed on focus change · on cluster move
04 Now / Next Execution shortlist with PRDs, approvals, and implementation packets. 1 — 5bets human or policy-gated promotion only
Signals stay raw · clusters stay canonical · the roadmap is a projection, not the storage layer

VII · Refactor
generateRoadmap()
Refactor · roadmap synthesis

The default action becomes attach, not mint.

Today · V1

Signals straight to briefs.
  1. Fetch unprocessed signals.
  2. Deduplicate.
  3. Summarize.
  4. Ask Claude to create ranked items.
  5. Mark signals processed.

V1.1 · After

Resolve · file · compute · refresh.
  1. Resolve current_focus + required pages.
  2. Run the filing resolver on fresh signals — attach, mint, or mark page stale.
  3. Compute evidence strength, freshness, confidence, effort, focus weighting — in code.
  4. Invoke roadmap-synthesis only on changed or high-uncertainty clusters.
  5. Refresh cluster briefs.
  6. Refresh the ranked roadmap slice.
  7. Write brain_runs.
  8. Update changed pages and citations.
Key policy

The default action for a new signal is to update an existing cluster, not to create a new brief.

Latent judgment stays latent. Arithmetic stays deterministic.

VIII · Refactor
generatePRD()
Refactor · PRD author

The PRD becomes a projection of memory, not a vacuum.

Today · V1

Evidence from the row itself.
  1. Load a roadmap row.
  2. Feed its evidence and traces back to Claude.
  3. Save prd_content on the roadmap row.

V1.1 · After

Repo-aware. Page-backed. Logged.
  1. Resolve overview · focus · repo_map · patterns · safety · metrics.
  2. Load the selected opportunity cluster + roadmap row.
  3. Load recent shipped changes that touch the same area.
  4. Invoke prd-author.
  5. Write brain_runs.
  6. Save prd_content.
  7. Update affected memory pages with new constraints, questions, and implementation notes.
Key policy

Guardrails belong in deterministic context — not improvised inside the prompt body.

Page-backed semantics · explicit citations · inspectable run log

IX · The Learning Loop
Ship → measure → update → propose
The learning loop.

Shipped work should feed back — into memory, and into the routing table.

STEP 01

Collect actual metrics.

After ship, pull the outcomes attached to the roadmap item and its experiment.

STEP 02

Compare to forecast.

Deterministic diff against the PRD's predicted deltas and success criteria.

STEP 03

Update cluster strength.

Adjust evidence strength, freshness, and dominant-need weighting based on the result.

STEP 04

Run impact-review.

Classify as confirmed, underperformed, or inconclusive. Explain the most likely reason.

STEP 05

Update pages.

Write the lesson back to the relevant project page with a citation to the ship.

STEP 06

Propose rule change.

If the same mistake is likely to recur, propose a skill, trigger, or resolver edit.

If the same judgment is needed twice, it should become a skill update, a filing rule, or a resolver rule — not a note someone remembers.

The roadmap system doesn't just learn what to build. It learns whether its own routing table is still accurate.

X · Dominant Need
current_focus as routing input
Focus modes.

A product only has one direction at a time. The backlog should know which.

Mode · 01 ux_quality Reduce friction, clarify flows, improve the felt quality of the product.
Raises
  • navigation clarity
  • error recovery
  • onboarding
  • dead-ends
Lowers
  • speculative bets
  • non-blocking infra
Mode · 02 conversion Improve visitor-to-signup or signup-to-paid in the current funnel.
Raises
  • landing clarity
  • pricing friction
  • CTA visibility
  • signup drop-off
Lowers
  • engagement loops w/o funnel tie
  • cosmetic polish
Mode · 03 virality Increase referral loops, sharing, and user-generated acquisition.
Raises
  • sharing surfaces
  • invite loops
  • social proof
  • network effects
Lowers
  • internal tooling
  • isolated retention fixes
Mode · 04 performance Reduce latency, errors, and operational drag that blocks quality or conversion.
Raises
  • load time
  • crash rate
  • render blocking
  • backend reliability
Lowers
  • nice-to-have feature work
  • long-tail content
Mode · 05 retention Increase repeat usage, habit formation, and long-term value after activation.
Raises
  • re-engagement hooks
  • time-to-value
  • churn reasons
  • unfinished loops
Lowers
  • top-of-funnel experiments
  • one-time launch mechanics
Ranking changes with focus mode · strategy isn't regenerated from scratch every run
